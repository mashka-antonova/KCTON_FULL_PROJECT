Что мне нужно

I.  модуль мониторинга

На уровне описания:

1. Система фильтрации (выбор субъекта, муниципалитета и диапазона лет)

Центральный блок управления данными, обеспечивающий выборку информации.
(значения по дефолту + изменение)

-   Субъект РФ: Выпадающий список всех регионов России, присутствующих в
    базе данных, отсортированный строго по алфавиту.

-   Муниципальное образование (МО): Каскадный фильтр. При выборе
    конкретного региона список МО автоматически ограничивается только
    теми районами и округами, которые входят в его состав. Если регион
    не выбран, доступен полный список всех МО в алфавитном порядке.

-   Период анализа: Инструмент выбора временного диапазона («Год начала»
    — «Год конца»).

    -   Валидация: Система динамически подгружает только те года, за
        которые в базе присутствуют полные данные.

    -   Логика: Выбор недоступных периодов блокируется для
        предотвращения ошибок.

2. Кнопка действия «Показать»

Инициализирует процесс агрегации данных и обновления визуальных
элементов. До нажатия кнопки дашборд отображает общую статистику по всей
стране (состояние по умолчанию).

3. Интерактивные демографические метрики (KPI) (есть значения по дефолту, изменяются при нажатии кнопки показать)

При активации фильтра отображается блок из 5 информационных карточек:

-   Численность населения: Текущее значение на выбранный конечный год.

-   Общее изменение: Разница между начальным и конечным годом
    (абсолютное значение и относительный рост/убыль в %).

-   Коэффициент рождаемости: Среднее значение за период.

-   Коэффициент смертности: Среднее значение за период.

-   Естественный прирост и Миграция: Итоговые показатели движения
    населения.

4. Визуализация: Тепловая карта плотности

Географический модуль для наглядной оценки распределения населения. (в
начале отображается)

-   Механика: Карта раскрашивается градиентом в зависимости от плотности
    населения в выбранных муниципалитетах.

-   Интерактивность: При наведении курсора (Hover) на область МО
    появляется всплывающая подсказка с данными:

    -   Название муниципалитета;

    -   Плотность населения (чел/км^2);

    -   Общая численность жителей.

5. Аналитические таблицы динамики

Два статических списка, отображающих экстремальные значения по стране
или выбранному региону(отображаются в самом начале):

-   Топ-10 лидеров роста: Муниципалитеты с самым высоким положительным
    процентом изменения населения.

-   Топ-10 лидеров убыли: Муниципалитеты с наибольшим сокращением
    численности населения.

-   Состав данных: Каждая строка таблицы включает название МО,
    актуальную численность и динамику в %.

На уровне реализации(могу быть не совсем точной, уточнить если что):

все параметры локаций (region_id, mo_id, mo_type) в запросах за данными
являются необязательными (optional).

-   Если я передаю эти параметры — фильтруешь данные (условие WHERE).

-   Если я их НЕ передаю — агрегируешь данные по всей России и
    возвращаешь общую статистику.

Ошибки

1. Ошибки валидации (400 Bad Request)

Это ситуации, когда пользователь выбрал в интерфейсе что-то логически
невозможное.

-   причина: Выбран «Год начала» больше, чем «Год конца» (например,
    старт в 2024, конец в 2020).

-   сообщение бэка: {"error": "Начальный год не может быть больше
    конечного"}.

-   реакция: Вывести предупреждение (Tooltip или Toast) и заблокировать
    кнопку «Показать».

2. Данные не найдены (404 Not Found)

Самая частая ситуация в демографии, так как официальная статистика часто
запаздывает на 1–2 года.

-   Причина: Запрошены данные за год, который еще не загружен в базу
    (например, за 2025 год).

-   Сообщение бэка: {"error": "Статистические данные за выбранный период
    отсутствуют"}.

-   реакция: Показать на месте графиков и карточек статус «Данные
    отсутствуют» (Empty State).

3. Частичное отсутствие данных (206 Partial Content)

-   Причина: По муниципалитету есть данные о населении, но нет данных о
    миграции или рождаемости за этот конкретный год.

-   Решение: Бэкенд присылает JSON, где вместо числа стоит null.

-   реакция: В карточке показателя вместо цифры выводишь прочерк «—» или
    текст «н/д» (нет данных). Это лучше, чем выводить «0», так как ноль
    — это тоже статистический показатель.

4. Ошибки картографии (500 Server Error)

Поскольку используется PostGIS для отрисовки тепловой карты, здесь могут
быть технические сбои.

-   Причина: Ошибка при генерации GeoJSON или слишком сложная геометрия,
    которую сервер не смог обработать.

-   Твоя реакция: На месте карты вывести сообщение: «Ошибка загрузки
    карты. Попробуйте обновить страницу или выбрать другой регион».

II. модуль прогнозирования

На уровне описания:

1. Система выбора территории (Фильтры)

Для удобства пользователя мы сохраняем преемственность с первой
страницей.

-   Иерархия: Каскадный выбор «Субъект РФ» $\rightarrow$ «Муниципальное
    образование».

-   Сортировка: Все списки строго по алфавиту.

-   Default State (Состояние по умолчанию): Чтобы избежать пустого
    экрана, при загрузке автоматически выбирается первый регион и его
    первый муниципалитет в списке, для которых сразу строится базовый
    прогноз.

2. Выбор ML-модели и параметров

-   Модель: На текущем этапе отображается название используемой модели
    (например, SARIMA или Prophet ). Интерфейс спроектирован так, чтобы
    в будущем можно было добавить выпадающий список для сравнения разных
    алгоритмов.

-   Горизонт прогноза: Выпадающий список с выбором периода: от 5 до 15
    лет (согласно ТЗ).

3. Интерактивный график временного ряда

Основной элемент страницы, визуализирующий демографическое будущее.

-   Исторические данные: Сплошная линия, отображающая реальную
    численность населения за все доступные прошлые годы.

-   Прогноз: Пунктирная линия, продолжающая график в будущее на
    выбранный горизонт.

-   Доверительный интервал (Облако): Полупрозрачная заливка вокруг линии
    прогноза, показывающая верхнюю и нижнюю границы (оптимистичный и
    пессимистичный сценарии).

4. Метрики качества и результаты

Блок из трех карточек, который появляется/обновляется после нажатия
кнопки «Рассчитать»:

-   MAPE (Средняя абсолютная ошибка в процентах);

-   RMSE (Среднеквадратичная ошибка);

-   MAE (Средняя абсолютная ошибка).

-   Примечание: Индикатор «Модель обучена» уберу скорее всего

На уровне реализации(могу быть не совсем точной, уточнить если что):

Общая логика работы страницы (Default State)

При первой загрузке страницы фронтенд запрашивает у бэкенда дефолтные
данные (например, берет первый по алфавиту регион и его первый
муниципалитет). График не должен быть пустым. При изменении параметров
(модели, горизонта, муниципалитета) и нажатии кнопки «Рассчитать»,
фронтенд показывает лоадер (крутилку) и отправляет новый запрос.

Фильтры локаций (регионы и муниципалитеты) переиспользуются из Модуля 1
(GET /api/filters/regions и GET /api/filters/municipalities).

Эндпоинт 1: Конфигурация прогноза (Параметры)

Чтобы фронтенд не «зашивал» названия моделей в код, бэкенд должен
отдавать список доступных алгоритмов. Либо если будет 1 модель, просто
текстовое поле с ее названием сделаю, тогда не надо передавать, но для
расширения в будущем как будто можно так попробовать(короче на
усмотрение)

-   URL: GET /api/forecast/config

-   Назначение: Получить список доступных ML-моделей и лимиты горизонта.

-   Ответ (JSON - 200 OK):

[]

Эндпоинт 2: Расчет прогноза (Core Endpoint)

Бэкенд вытаскивает исторические данные, прогоняет их через ML-модель на
лету (или достает заранее посчитанное из БД) и возвращает результат
вместе с метриками.

-   URL: GET /api/forecast/calculate

-   Query-параметры (Все обязательные):

    -   mo_id (integer) — ID муниципалитета.

    -   horizon (integer) — Горизонт прогноза (от 5 до 15).

    -   model (string) — Идентификатор модели (например, prophet).

💡 В библиотеках графиков (типа Recharts) линия строится по массиву точек. Если бэкенд пришлет исторические данные и прогноз в разных массивах, на графике между «Вчера» и «Завтра» будет визуальный разрыв.

Чтобы линия была непрерывной, последняя точка истории (например, 2026
год) должна являться первой точкой прогноза.

-   Ответ (JSON - 200 OK):

[]

fact: для отрисовки <Line dataKey="fact" stroke="#00008B" /> (Сплошная
темно-синяя линия).

forecast: для <Line dataKey="forecast" stroke="#1E90FF"
strokeDasharray="5 5" /> (Пунктирная ярко-синяя линия).

low и high: для компонента <Area dataKey="high" ... /> (Полупрозрачная
заливка «облака» доверительного интервала). Отрисовываешь площадь от low
до high.

Обработка ошибок

400 Bad Request — Некорректные параметры.

422 Unprocessable Entity — Недостаточно данных для ML.

-   Причина: В базе есть данные по этому муниципалитету только за 2
    года. Модель не может обучиться на таком коротком отрезке.

-   Ответ бэка: {"error": "Недостаточно исторических данных для
    построения прогноза по выбранному МО"}.

-   реакция: Скрыть график, показать красивую заглушку: «Для данного
    муниципалитета прогноз недоступен».

500 Internal Server Error — Упала Python-модель.

-   Причина: Ошибка в математике (например, деление на ноль в библиотеке
    Prophet).

-   реакция: Снять лоадер (крутилку) и написать «Ошибка сервера при
    расчете модели».

! Гоша, обрати внимание на массив chart_data. Я специально сделала его
плоским (одним списком), а не разбила на два разных массива для истории
и будущего. Это критически важно для фронтенда, чтобы библиотека
Recharts смогла плавно соединить историческую линию с линией прогноза и
нарисовать вокруг нее тени доверительного интервала.

III. модуль Аналитической справки

На уровне описания:

1.  Выбор муниципалитета

Для сохранения преемственности интерфейса используется такая же механика
фильтрации как в первых 2х модулях:

-   Выбор субъекта РФ: Список всех регионов в алфавитном порядке.

-   Выбор муниципального образования: Зависимый список, отображающий
    только те МО, которые входят в выбранный субъект. Также строго
    отсортирован по алфавиту.

-   Состояние по умолчанию: При открытии модуля активны те же регион и
    МО, которые были выбраны на предыдущих вкладках (сквозная
    фильтрация).

Настройка параметров анализа

Пользователь определяет «глубину» погружения ИИ в данные:

-   Горизонт прогнозирования: Выбор периода, на который ИИ будет строить
    качественную оценку (на текущий момент — фиксированные кнопки «На 5
    лет» и «На 10 лет»), в дальнейшем возможно уставнавлю возможность
    выбрать другие значения

  3. Интеллектуальная генерация отчета

Центральный функционал модуля, использующий интеграцию с большой
языковой моделью (LLM).

-   Процесс: После нажатия кнопки «Сформировать справку» система
    собирает данные из модулей мониторинга и прогноза и отправляет их
    ИИ-агенту.

-   Содержание справки: ИИ формирует структурированный текст, включающий
    краткое резюме динамики населения, анализ тенденций (например,
    причины убыли или роста), оценку прогноза и конкретные рекомендации
    по развитию социальной политики региона

  4. Интерактивный предпросмотр и экспорт

Результат работы ИИ отображается в специальном окне, имитирующем лист
документа.

-   Preview-зона: Позволяет ознакомиться с текстом справки сразу в
    веб-приложении, не скачивая файл.

-   Многоформатный экспорт: Интегрированные кнопки для мгновенной
    выгрузки сформированного текста в форматах PDF или Word (docx).

На уровне реализации(могу быть не совсем точной, уточнить если что):

Блок 1: Эндпоинты управления (Используются фильтры из Модуля 1)

Для выбора территории используются те же эндпоинты: GET
/api/filters/regions и GET /api/filters/municipalities.

3. Обработка ошибок

1.  429 Too Many Requests (Лимит ИИ):

    -   Причина: Закончились токены или превышен лимит запросов к API
        нейросети.

    -   реакция: «Сервис генерации перегружен, попробуйте через минуту».

2.  504 Gateway Timeout (Долгое ожидание):

    -   Причина: Нейросеть «зависла» и не ответила вовремя.

    -   реакция: Прекратить опрос статуса и предложить пользователю
        нажать кнопку повторно.

3.  Ошибка данных:

    -   Причина: В базе нет данных по этому МО, ИИ не на чем строить
        отчет.

    -   реакция: Не давать нажать кнопку «Сформировать», если данные в
        Модуле 1 и 2 пустые.
