Задача 1. Мониторинг численности населения

Разработать веб-приложение с мониторингом численности населения и
демографических показателей по муниципальным образованиям России.
Приложение должно обеспечивать сбор данных из открытых источников
(включая данные Росстат, ЕМИСС, муниципальные порталы и иные публичные
источники).

Интерфейс должен отображать:

-   Численность населения по муниципалитетам (городские округа,
    муниципальные районы);

-   Динамику изменения населения за выбранный период (рост/снижение в
    %);

-   Ключевые демографические показатели (коэффициент рождаемости,
    смертности, естественный прирост, миграция);

-   Таблицу муниципалитетов с наибольшим ростом/снижением населения;

-   Тепловую карту по регионам с градацией плотности населения.

Должна быть реализована фильтрация по субъекту РФ, типу муниципального
образования, периоду анализа.

План реализации Задачи 1: Мониторинг численности населения

Шаг 1: Сбор и очистка данных (Data Engineering)

-   Источники: Написать парсеры (Python + requests + BeautifulSoup) для
    скачивания таблиц с Росстата и скрипты для работы с API ЕМИСС.

-   Очистка (Pandas): Привести названия муниципалитетов к единому
    стандарту (чтобы "г. Казань" и "городской округ Казань" склеились в
    одно). Обработать пропущенные значения (например, заполнить средним
    или перенести с предыдущего года).

-   Геоданные: Найти и скачать границы муниципалитетов (GeoJSON) и
    связать их с кодами ОКТМО.

Шаг 2: База данных (Backend & DB)

-   Хранилище: Поднять PostgreSQL. Обязательно установить расширение
    PostGIS, так как вам придется работать с геометрией (границами
    районов) для тепловой карты.

-   Структура: Создать таблицы:

  --------------------------------------------------------------------------
  Таблица               Ключевые поля                Описание
  --------------------- ---------------------------- -----------------------
  regions               id, name, federal_district,  Субъекты РФ
                        oktmo_code                   

  municipalities        id, region_id, name,         Муниципальные
                        mo_type, oktmo_code,         образования с
                        geom(PostGIS)                геометрией

  population_history    mo_id, year, population,     Исторические данные (из
                        birth_rate, death_rate,      ETL)
                        natural_growth, migration,   
                        source                       

  population_forecast   mo_id, forecast_year,        Прогнозные значения
                        horizon_years, value_low,    
                        value_mid, value_high,       
                        model_name, created_at       

  model_metrics         mo_id, model_name, mape,     Качество ML-моделей
                        rmse, mae, r2, train_period, 
                        test_period                  
  --------------------------------------------------------------------------

Шаг 3: Разработка API (Backend)

-   Написать эндпоинты (например, на FastAPI):

    -   GET /api/municipalities — список с возможностью фильтрации.

    -   GET /api/stats — выдача статистики для графиков.

    -   GET /api/heatmap — выдача файла GeoJSON, где к каждому полигону
        приклеен показатель плотности населения.

Шаг 4: Интерфейс и Визуализация (Frontend)

-   Дашборд и Фильтры: Сверстать панель управления (выбор региона, типа
    МО, диапазона дат).

-   Графики: Использовать библиотеки вроде Recharts или ECharts для
    построения линейных графиков динамики и столбчатых диаграмм
    (рождаемость/смертность).

-   Карта: Подключить библиотеку Leaflet или Mapbox для рендера GeoJSON
    и раскраски регионов (создание тепловой карты).

Задача 2. Прогнозирование численности населения

Реализовать модуль прогнозирования численности населения на основе
исторических данных.

Система должна:

-   Формировать прогноз на горизонт от 5 до 15 лет;

-   Отображать на графике фактические данные и прогнозные значения;

-   Показывать прогнозный интервал;

-   Рассчитывать метрики качества модели (MAPE, RMSE, MAE).

Допускается использование любых моделей анализа временных рядов и
методов машинного обучения.

План реализации Задачи 2: Прогнозирование численности населения

Шаг 1: Подготовка временных рядов (Data Science)

-   Выгрузить из БД очищенные исторические данные по каждому
    муниципалитету (минимум за 10-15 лет).

-   Проверить ряды на аномалии (резкие скачки из-за изменения границ
    муниципалитетов) и сгладить их.

Шаг 2: Выбор и обучение ML-моделей // оставьте мне пж это (Анель)

-   Базовый уровень: Использовать библиотеку Prophet (от Meta). Она
    отлично работает с временными рядами и сама строит прогнозные
    интервалы.

-   Альтернатива: Использовать модели ARIMA/SARIMA или градиентный
    бустинг (LightGBM), добавив внешние факторы (экономические
    показатели).

-   Написать скрипт, который прогоняет модель для каждого муниципалитета
    и сохраняет прогноз на 5, 10, 15 лет в базу данных.

Шаг 3: Расчет метрик качества

-   Разделить исторические данные на тренировочную (например, 2010–2020)
    и тестовую (2021–2023) выборки.

-   Сделать прогноз на тестовый период и сравнить его с фактом,
    рассчитав метрики MAPE, RMSE и MAE.

Шаг 4: Вывод прогноза на Frontend

-   Обновить графики: исторические данные отображать сплошной линией,
    прогноз — пунктирной.

-   Отрисовать "прогнозный интервал" (доверительную зону) в виде
    полупрозрачной заливки вокруг пунктирной линии.

-   Вывести карточки с метриками качества модели.

Задача 3. Генерация аналитической справки с ИИ

Реализовать модуль автоматического формирования аналитической справки на
основе данных мониторинга и прогноза.

Справка должна содержать:

-   Краткое резюме динамики населения по муниципалитету/региону;

-   Выявленные демографические тенденции и возможные факторы влияния;

-   Прогнозную оценку на 5-10 лет;

-   Рекомендации по социальной политике и территориальному планированию.

Формирование отчёта может быть реализовано через шаблонизатор либо
интеграцию с LLM. В интерфейсе должна быть предусмотрена кнопка
формирования и выгрузки отчёта (PDF/Word).

Шаг 1: Проектирование структуры справки и источников данных

· Определить фиксированные разделы документа: титульный лист (название
МО/региона, период анализа), введение, динамика населения,
демографические показатели, прогноз, выводы и рекомендации.

· Спроектировать, какие данные из БД (PostgreSQL) подтягиваются в каждый
раздел:

· численность населения по годам, темпы роста/убыли;

· коэффициенты рождаемости, смертности, миграции;

· прогнозные значения с доверительными интервалами (из таблицы
population_forecast);

· метрики качества модели (из таблицы model_metrics).

· Определить формат выгрузки: PDF (через библиотеку генерации отчётов)
и/или DOCX (через шаблон с подстановкой).

Шаг 2: Разработка шаблонизированного отчёта (без LLM – базовый вариант)

· Создать шаблон документа в формате HTML+CSS (для последующей
конвертации в PDF через wkhtmltopdf или WeasyPrint) или в формате .docx
с полями-заменителями (например, {{population_dynamics}}).

· Написать скрипт на Python (например, с использованием Jinja2 для
текстовых шаблонов и python-docx или reportlab для офисных форматов),
который:

· выполняет запросы к API бэкенда (или напрямую к БД) за нужными
данными;

· подставляет числовые значения, таблицы и простые текстовые выводы
(например, «наблюдается убыль населения со средним темпом –1.2% в год»);

· генерирует готовый файл.

Шаг 3: Интеграция с LLM для формирования текстовых выводов и
рекомендаций

· Выбрать LLM (например, GPT через API, или локальную модель типа Llama
2 / Qwen, если требуется конфиденциальность).

· Создать промпты, которые на вход получают:

· статистику по муниципалитету (ряды данных, прогноз, метрики);

· краткий контекст (регион, тип МО).

· LLM должна вернуть:

· резюме динамики;

· перечень выявленных тенденций с возможными факторами (например, отток
молодёжи, старение населения);

· прогнозную оценку простыми словами;

· 3–5 конкретных рекомендаций для администрации.

· Реализовать вызов LLM из модуля генерации отчёта. Предусмотреть
кэширование результатов для одних и тех же параметров отчёта, чтобы не
тратить токены при повторных запросах.

Шаг 4: Разработка API и привязка к интерфейсу

· Создать эндпоинт (например, POST /api/generate_report), принимающий
параметры:

· municipality_id (или region_id);

· report_format (pdf / docx);

· forecast_horizon (5 или 10 лет).

· Эндпоинт запускает асинхронную задачу генерации отчёта (чтобы не
блокировать интерфейс).

· В ответ возвращает task_id, по которому фронтенд может опрашивать
статус и получать ссылку на готовый файл.

Шаг 5: Фронтенд – кнопка выгрузки и превью

· Добавить в интерфейс (рядом с фильтрами или на карточке
муниципалитета) кнопку «Сформировать аналитическую справку».

· При клике показывать модальное окно с выбором формата (PDF/Word) и
горизонта прогноза.

· Отображать индикатор загрузки (процесс генерации может занимать
несколько секунд из-за LLM).

· После готовности предложить скачать файл или открыть его в новой
вкладке.

· (Опционально) добавить предпросмотр справки прямо в браузере перед
скачиванием.
